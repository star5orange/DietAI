## 术语表



| 术语 | 通俗解释 | 在本项目中的作用 |
| --- | --- | --- |
| **Agent** | 这里指智能体，可以自主调用工具、记忆上下文、做出决策的 AI 程序 | 本项目有 5 个 Agent：营养分析、增强营养、聊天、目标追踪、深度营养师 |
| **APScheduler** | Python 定时任务库，帮你设定"每天凌晨 2 点自动做某事" | 每天自动更新用户记忆文件、每周生成周报 |
| **AsyncValue** | Riverpod 的数据包装器，有三种状态：loading(加载中)、data(有数据)、error(出错了) | UI 根据三种状态显示不同的界面（loading 圈、数据列表、错误提示） |
| **bcrypt** | 一种密码哈希算法，把密码变成不可逆的乱码，即使数据库泄露也看不出原文 | 用户密码存储方式 |
| **ChromaDB** | 一个向量数据库，存储知识的"语义向量"，可以根据意思搜索而不是关键词匹配 | 存储营养知识库，支持按语义检索 |
| **Deep Agents SDK** | 一个高级 AI Agent 开发框架，内置记忆、工具、技能、子 Agent 机制 | 构建 diet_deep_agent（私人营养师 AI）的底层框架 |
| **FastAPI** | 一个 Python Web 框架，帮你快速搭建后端 API 服务 | 后端服务框架，处理所有 HTTP 请求 |
| **fire_and_forget** | "发了就不管"，不等结果就继续往下执行，类似发微信消息不用等回复 | 用户操作完成后，异步更新记忆文件，不阻塞响应 |
| **Flutter** | Google 的跨平台移动端开发框架，一套代码跑 Android 和 iOS | 移动端 APP 开发框架 |
| **GoRouter** | Flutter 的路由库，帮你管理页面跳转和"哪些页面需要登录才能看" | 前端路由导航和认证守卫 |
| **JWT** | JSON Web Token，一串加密字符串，用来证明"我是谁" | 用户登录后拿到 token，后续请求带上 token 就不用重复登录 |
| **LangGraph** | 一个 Python 框架，帮你把 AI 的大脑拆成多个步骤（节点），按顺序执行 | AI Agent 的执行引擎，定义"先做什么再做什么" |
| **LangGraph SDK** | 通过 HTTP 调用 LangGraph Agent 服务的客户端库 | 后端通过 SDK 调用 Agent 服务（端口 2024） |
| **Markdown 记忆** | 用 Markdown 文件存 AI 的"记忆"，类似于给 AI 一本笔记本 | AI 每次对话前先读记忆文件，记住用户偏好和过去的对话 |
| **ORM** | Object-Relational Mapping，用代码对象操作数据库，而不是写 SQL 语句 | 用 SQLAlchemy 操作 PostgreSQL |
| **PoW** | Proof of Work，工作量证明 | 本项目未使用 |
| **Provider** | Riverpod 的依赖注入和状态管理单位 | 服务的创建和数据的获取都通过 Provider |
| **RAG** | Retrieval-Augmented Generation，先从知识库检索相关内容，再让 AI 基于检索结果回答 | AI 回答营养问题时，先从向量库检索相关知识，再生成回答 |
| **Riverpod** | Flutter 的状态管理库，帮你管理 App 中"数据变了界面就刷新"的逻辑 | 前端状态管理，替代传统的 setState |
| **SessionLocal** | SQLAlchemy 创建数据库连接的方式，每次请求创建一个连接，请求结束自动关闭 | 在路由中通过 `get_db()` 依赖注入获取 |
| **SSE** | Server-Sent Events，服务端主动推数据给客户端，类似于"服务器一边想一边说" | AI 分析结果逐步推送给前端，让用户看到"正在分析中..."而不是等很久 |
| **SSE 流式** | 一种通信方式，服务器逐步发数据，客户端逐步显示，用户不用等到全部完成 | 拍照识别食物时，逐步显示"正在分析图片→正在提取营养→正在生成建议" |
| **StateNotifier** | Riverpod 中管理状态的对象，持有数据并在数据变化时通知 UI 刷新 | 每个 Provider 绑定一个 StateNotifier |
| **SyncService** | 同步服务，把数据库中的数据"翻译"成 Markdown 记忆文件给 AI 读 | 数据库数据变了 → 自动更新记忆文件 |

## 一、系统架构总览

### 1.1 系统定位

> **大白话**：DietAI 就像一个"智能营养师 APP"。你拍照上传食物，它告诉你有多少卡路里；你记录饮食，它帮你计算营养是否均衡；你定一个减肥目标，它告诉你每天该怎么吃。

DietAI 是一款面向中国用户的智能饮食健康管理应用。系统采用**三层架构**：

1. **移动端 APP（Flutter）**—— 用户看到的界面，负责交互和数据展示
2. **后端服务（FastAPI）**—— 处理业务逻辑、数据存储、AI 调度
3. **AI Agent 服务（LangGraph）**—— 提供 AI 分析能力（营养识别、饮食建议、目标追踪等）

核心能力包括：
- 📸 拍照识别食物，自动计算热量和营养素
- 🤖 AI 个性化饮食建议（结合用户偏好、历史饮食、健康目标）
- 🎯 健康目标追踪（减脂/增肌/维持体重）
- 💬 AI 对话咨询（营养咨询、健康评估、运动建议）

### 1.2 架构风格

> **设计理由**：为什么选这些技术？

| 维度 | 选型 | 为什么这样选 |
| --- | --- | --- |
| 整体架构 | 前后端分离 + AI 微服务 | 前端和后端独立开发部署，AI 服务可以独立扩容，互不影响 |
| 后端架构 | 分层架构 + 事件驱动 | 代码按路由→服务→模型分层，清晰好维护；数据变更后异步更新 AI 记忆，不阻塞用户请求 |
| 前端架构 | Feature-First + 状态驱动 | 按功能模块组织代码，每个功能自包含；用 Riverpod 统一管理数据状态，界面自动刷新 |
| AI 架构 | 多 Agent 图编排 | 不同任务交给不同 Agent（营养分析、聊天、目标追踪），各司其职；Deep Agent 作为"总调度"，自主选择合适的工具 |
| 数据存储 | PostgreSQL + MinIO + Redis + ChromaDB + Markdown | 图片存 MinIO，缓存用 Redis，知识检索用 ChromaDB，AI 记忆用 Markdown |
| 通信方式 | REST + SSE | 常规增删改查用 REST（一问一答）；AI 分析用 SSE（服务器逐步推送结果，用户不必等全部完成） |

**架构图解读**（从上往下看数据是怎么流的）：

1. 用户在 APP 上操作（比如拍了张食物照片）
2. APP 通过 HTTP 把图片发给后端
3. 后端收到请求，把图片存到 MinIO 对象存储，创建一条食物记录存入 PostgreSQL
4. 后端通过 LangGraph SDK 把图片和用户偏好发给 AI Agent 服务
5. AI Agent 从 ChromaDB 检索营养知识，从记忆文件读取用户偏好，调用大模型生成营养分析
6. AI 分析结果通过 SSE 逐步推回给 APP，用户看到"正在分析→营养数据→建议"
7. 后端把分析结果存到 PostgreSQL，同时异步更新 AI 记忆文件
8. 用户下次再来时，AI 已经"记住"了之前的偏好和习惯

## 二、后端架构

### 2.1 分层架构

> **大白话**：后端代码像千层蛋糕一样分层，请求从最上面进来，一层层往下走，每层只做自己的事。

```
┌─────────────────────────────────────────────────────────────┐
│  🌐 路由层（Router）—— "接待员"：决定请求该交给谁处理          │
│     9个模块：认证│用户│食物│健康│对话│分析│目标│深度│收藏       │
│     只做：参数校验、权限检查、调用服务、返回响应               │
├─────────────────────────────────────────────────────────────┤
│  🔧 服务层（Service）—— "业务核心"：真正干活的逻辑             │
│     AuthService(登录注册) │ AgentOrchestrator(调度AI)         │
│     CacheService(缓存) │ NutritionCalc(营养计算)               │
│     SyncService(记忆同步) │ MemoryManager(记忆读写)            │
│     只做：业务逻辑、数据组装、调用AI、触发事件                  │
├─────────────────────────────────────────────────────────────┤
│  📦 模型层（Model）—— "数据容器"：定义数据长什么样              │
│     SQLAlchemy ORM 模型（16张表）+ Pydantic Schema（请求/响应）  │
│     只做：数据结构定义、验证、序列化                           │
├─────────────────────────────────────────────────────────────┤
│  🏗️ 基础设施层（Infrastructure）—— "地基"：底层服务             │
│     PostgreSQL │ MinIO │ Redis │ ChromaDB                    │
│     APScheduler │ 事件处理器 │ 中间件 │ 认证                   │
│     只做：数据存储、缓存、文件存储、向量检索、定时任务           │
└─────────────────────────────────────────────────────────────┘
```

**为什么分层？**
- 路由层不关心业务逻辑，只管"这个请求该给谁"
- 服务层不关心数据怎么存，只管"这件事该怎么处理"
- 模型层不关心请求从哪来，只管"数据长什么样"
- 基础设施层不关心业务，只管"数据往哪里放"
- 分层后，改某一层不会影响其他层，比如换数据库只需改基础设施层

### 2.2 请求生命周期

> **大白话**：一个请求从进门到出门，经过了哪些关卡。

```
用户拍照上传食物图片
    │
    ▼
① 中间件链（保安检查）
    CORS检查 → 主机验证 → 记录处理时间 → 记录请求日志
    │
    ▼
② 路由分发（前台接待）
    food_router 收到 POST /api/foods/records
    │
    ▼
③ 认证检查（门禁卡验证）
    get_current_user 从 JWT token 中提取用户ID
    │
    ▼
④ 业务处理（专业处理）
    ├── 图片存入 MinIO，获取 URL
    ├── 创建 FoodRecord 记录到 PostgreSQL（保存图片 URL）
    ├── 调用 AgentOrchestrator.analyze_food_with_goals()
    │     ├── 读取用户记忆（从 MD 文件）
    │     ├── 计算用户目标（从数据库）
    │     └── 调用 AI 分析（通过 LangGraph SDK）
    ├── 保存 NutritionDetail 到 PostgreSQL
    ├── 更新 DailyNutritionSummary
    │
    ▼
⑤ 异步事件（后台通知）
    fire_and_forget(on_food_record_created)
    → 更新 AI 记忆文件（不阻塞用户响应）
    │
    ▼
⑥ 返回响应
    SSE 流式推送每步结果给 APP
```

**为什么用"发了就不管"（fire_and_forget）写记忆？**
用户等的是"分析结果"，记忆更新是后台任务。如果等记忆更新完再返回，用户会多等 1-2 秒，体验不好。所以记忆更新放在后台异步做，用户即刻得到响应。

### 2.4 认证架构

> **大白话**：就像酒店入住——注册拿到房卡，之后刷房卡进出。房卡有过期时间，快过期了可以用长卡换新短卡。

```
用户注册 → 密码变成乱码(bcrypt) → 存到数据库
                                          ↓
用户登录 → 验证密码 → 发两张卡:
           ├── 短卡(access_token): 30分钟有效，日常进出用它
           └── 长卡(refresh_token): 7天有效，用来换新短卡

每次请求 → 刷短卡 → 验证通过 → 放行
         → 短卡过期(401) → 拿长卡换新短卡 → 重新请求

APP端处理:
         → 登录成功 → 安全存储两张卡(FlutterSecureStorage)
         → 任何请求带上短卡(Authorization: Bearer <token>)
         → 遇到401 → 自动拿长卡换新短卡 → 重试原请求
         → 换卡也失败 → 清空卡 → 跳转到登录页
```

**为什么用双令牌？**
如果只用一个长有效期 Token，一旦泄露，攻击者可以长期冒充用户。双令牌机制让短 Token 很快过期（30分钟），即使泄露也影响有限；长 Token 只用来换短 Token，不直接访问数据，风险更可控。

### 2.5 数据库架构

#### 2.5.1 核心表关系

> **大白话**：数据库里的表就像 Excel 文件，每张表存一种数据。表之间通过"外键"关联——就像学生表和班级表之间通过"班级编号"关联一样。

```
用户(User) 是核心，其他表都围着它转：

User(用户) ──1:1── UserProfile(用户资料)
  │                    存性别、身高、体重、活动水平等
  │
  ├──1:N── HealthGoal(健康目标)
  │                    "我要减脂"、"我要增肌"
  │
  ├──1:N── Disease(疾病)
  │                    糖尿病、高血压等
  │
  ├──1:N── Allergy(过敏)
  │                    海鲜过敏、花生过敏等
  │
  ├──1:N── WeightRecord(体重记录)
  │                    每次量体重的记录（体重、体脂率、肌肉量）
  │
  ├──1:N── FoodRecord(食物记录) ──1:1── NutritionDetail(营养详情)
  │     每次吃了什么          16项营养素详细数据
  │
  ├──1:N── DailyNutritionSummary(每日汇总)
  │     一天的总热量、蛋白质、脂肪等汇总
  │
  ├──1:N── ConversationSession(对话会话) ──1:N── ConversationMessage(对话消息)
  │     和AI的一次对话                       对话中的每条消息
  │
  ├──1:N── SavedMeal(收藏菜品) ──1:1── SavedMealNutrition(菜品营养)
  │     用户保存的常用菜品                   每份菜品的营养数据
  │
  └──1:N── UserSavedMealFavorite(收藏关系)
                哪个用户收藏了哪个菜品
```

#### 2.5.3 数据库连接配置

| 配置 | 值 | 说明 |
| --- | --- | --- |
| 引擎 | PostgreSQL 15 | 功能最全面的开源数据库 |
| 连接池大小 | 20 | 同时最多 20 个数据库连接 |
| 连接回收 | 300秒 | 每 5 分钟回收空闲连接，防止连接泄漏 |
| 预检查 | pool_pre_ping=True | 每次使用前检查连接是否活着的 |
| 隔离级别 | READ_COMMITTED | 读不加锁，写只锁修改的行 |
| 建表方式 | 启动时 `create_all()` | 应用启动时自动创建不存在的表 |

**为什么不用数据库迁移工具？**
项目初期为了快速开发，选用了 `create_all()` 自动建表。但这种方式只能创建新表，不能修改已有表的结构（比如加列、改类型）。后续如果需要改表结构，就需要引入 Alembic 迁移工具。

### 2.6 图片存储

> **大白话**：用户拍的食物图片存到独立的 MinIO 对象存储服务中，数据库只保存图片的 URL。


**图片存储流程：**
```
用户拍照/选图 → APP 压缩图片 → base64 编码 → POST 到后端
    → 后端存入 MinIO 对象存储服务
    → 后端将返回的图片 URL 存入 PostgreSQL 的 food_records 表
    → 需要显示图片时：前端直接通过 URL 访问 MinIO 获取图片
```

### 2.7 数据缓存策略

> **大白话**：本项目使用 Redis 缓存服务，加速高频查询响应，并支持限流、会话管理等功能。



**前端三层缓存（仍然保留）：**

```
请求图片数据：
  │
  ├── 第1层：内存缓存（钱包，最快）
  │     最多存 50 条图片数据，不过期
  │     找到了？直接返回！
  │
  ├── 第2层：SharedPreferences（抽屉，稍慢）
  │     JSON 格式存，24小时过期
  │     找到了？先放进内存缓存，再返回
  │
  ├── 第3层：文件缓存（银行，最慢但最全）
  │     图片 base64 解码后存到临时目录
  │     不过期（依赖系统清理）
  │     找到了？先放进内存缓存，再返回
  │
  └── 都没有？→ 从服务器 API 获取
             → 存入内存缓存 + SharedPreferences + 文件缓存
             → 返回数据
```

**AI 知识检索缓存（ChromaDB 内部处理）：**

- RAG 检索结果由 ChromaDB 自身的查询机制返回
- 前端没有额外的缓存层，每次 AI 分析直接调用 LangGraph 服务

### 2.8 定时任务架构

> **大白话**：就像闹钟一样，系统会在固定时间自动做某些事，不需要人工干预。

| 什么时候 | 做什么 | 为什么 |
| --- | --- | --- |
| 每天凌晨 2:00 | 更新最近 7 天活跃用户的共享记忆 | 用户资料可能改了，AI 需要知道最新信息 |
| 每天凌晨 2:30 | 更新有活跃目标的用户的目标追踪记忆 | 每天的摄入变了，目标追踪数据需要更新 |
| 每周日 3:00 | 生成活跃用户的营养周报 | 总结这周吃了什么，AI 下周能给更好的建议 |
| 每周日 3:30 | 生成活跃用户的对话摘要 | AI 需要知道这周聊了什么话题 |

### 2.9 事件驱动架构

> **大白话**：用户做了某件事（比如记录了一顿饭），系统会自动通知 AI:"嘿，这个用户的数据变了，你该更新记忆了"。这就像你更新了个人信息后，银行自动给你发新账单一样。

```
用户操作（触发事件）              系统后台自动做的事
─────────────────              ─────────────────
记录了一顿饭 ─────────────→ 更新营养工作区 + 更新目标追踪记忆
记录了体重 ────────────────→ 更新目标追踪 + 更新共享记忆
改了健康目标 ──────────────→ 重算每日目标 + 更新目标追踪
结束了对话 ────────────────→ 更新对话工作区（话题和互动摘要）
改了个人资料 ──────────────→ 更新共享记忆 +（如果体重/身高变了）更新目标追踪
改了过敏/疾病 ─────────────→ 更新共享记忆（AI 要知道这个）
```

**一致性问题**：这些更新是异步的（"发了就不管"），所以 AI 记忆可能短暂落后于数据库（最多几秒）。定时任务作为兜底，每天凌晨全量同步一次，确保不会长期不一致。

## 三、AI Agent 架构

### 3.1 Agent 总览

> **大白话**：本项目有 4个 AI 智能体（Agent），就像一个营养师团队里的不同角色：

```
┌─────────────────────────────────────────────────────────┐
│          🤖 DietAI 智能体团队（运行在 LangGraph 服务上）    │
│                                                          │
│  🍎 nutrition_agent（营养分析师）                         │
│     擅长：看照片说食物成分、算热量、给建议               │
│     流程：拍照 → 识别 → 算营养 → 查知识库 → 给建议       │
│                                                          │

│                                                          │
│  💬 chat_agent（营养咨询师）                              │
│     擅长：和用户聊天，回答营养、健康、运动问题           │
│     4 种模式：营养咨询 / 健康评估 / 食物识别 / 运动建议   │
│                                                          │
│  🎯 goal_tracking_agent（目标监督员）                     │
│     擅长：算你每天该吃多少、离目标还差多远               │
│     流程：读用户数据 → 算基础代谢 → 算每日目标 → 评估进度 → 给建议 │
│                                                          │
│  🧠 diet_deep_agent（私人营养师·总调度）                  │
│     擅长：全能选手 —— 会聊天、会分析、会记住你的喜好     │
│     有 8 个工具（查数据/记体重/学偏好等）                 │
│     有 5 项技能（会话/营养分析/饮食规划/健康评估/模式检测） │
│     有长期记忆（越用越懂你）                              │
│     可以自己决定什么时候用什么工具                        │
│                                                          │

└─────────────────────────────────────────────────────────┘
```

**为什么需要这么多 Agent？**
每个 Agent 专注做一件事，职责清晰。nutrition_agent 只管分析照片，不会分心去聊天；chat_agent 只管聊天，不会去算营养。这样每个 Agent 更容易调试和优化。

而 **diet_deep_agent（私人营养师）** 像个经验丰富的**总营养师**——它什么都能做，可以自己决定什么时候调用什么工具。比如用户问"我今天还能吃什么"，它会自己查"今天吃了多少"（工具）+ "用户偏好是什么"（记忆）+ "还有什么健康知识"（RAG），然后给出综合建议。

其他 Agent 是"专线司机"（只跑一条线），它是"出租车司机"（根据目的地自己选路）。

### 3.2 nutrition_agent 工作流程

> **大白话**：营养分析师的工作就像一个营养师看一张食物照片，然后一步步分析。

```
用户上传食物照片
    │
    ▼
📷 state_init（准备阶段）
    把照片转成 base64 编码，初始化状态
    │
    ▼
🔍 analyze_image（看图说话）
    把照片交给视觉大模型（qwen-vl-max 或 gpt-4.1-nano）
    大模型说："这碗是红烧牛肉面，有面条、牛肉、葱花..."
    │
    ▼
📊 extract_nutrition（提取数据）
    把大模型的描述交给分析模型（qwen3-32b 或 o3-mini）
    让它按结构化格式输出：热量550kcal、蛋白质25g、脂肪18g...
    │
    ▼
📚 retrieve_knowledge（查知识库）
    去 ChromaDB 向量数据库搜索相关知识："牛肉面通常热量是多少？"
    ChromaDB 向量检索营养知识
    │
    ▼
🔗 generate_dependencies（生成依赖关系）
    把食物数据和知识库信息整理成"建议依赖"（指导下一步怎么给建议）
    │
    ▼
💡 generate_advice（生成建议）
    综合所有信息，生成个性化建议："这碗面热量偏高，建议晚餐吃清淡些..."
    │
    ▼
✅ format_response（格式化输出）
    返回最终结果
```

### 3.3 营养分析功能的个性化支持

> **大白话**：nutrition_agent 现在支持用户记忆功能，可以在分析食物时考虑用户的过敏和偏好，提供个性化建议。

```
nutrition_agent 原版流程:
  看照片 → 识别 → 提取营养 → 查知识 → 给建议

enhanced 版多了这两步:
    ↓ 在"给建议"之前
  🧠 load_shared_memory → 读取用户偏好、过敏、疾病信息
    ↓ 在"给建议"环节
  🎯 generate_advice_with_context → 过敏交叉检查 + 个性化建议
    ↓ 在"格式化"之前
  💾 save_to_nutrition_md → 把分析结果写回记忆文件

所以 enhanced 版会:
  ✅ 知道你对海鲜过敏，不会推荐虾
  ✅ 知道你最近一周都在吃高热量，会提醒你
  ✅ 记住这次分析的结果，下次能看到变化趋势
```

### 3.4 diet_deep_agent 架构详解

> **大白话**：这是最强大的 Agent，像一个经验丰富的私人营养师。它有工具箱、技能书、记忆本和一个小助手。

```
┌──────────────── diet_deep_agent 总调度中心 ────────────────┐
│                                                            │
│  🛠️ 工具箱(8个工具，可以随时调用)：                         │
│  ┌──────────────────────────────────────────────────────┐ │

│  │ 📋 lookup_food_database  → 查数据库里的食物记录          │ │
│  │ 📊 get_daily_status      → 查今天吃了多少、还差多少      │ │
│  │ 🧮 calculate_targets     → 计算每日应该吃多少            │ │
│  │ ⚖️ record_weight         → 记录体重                     │ │
│  │ 🔍 query_nutrition_knowledge → 从知识库搜索营养知识      │ │
│  │ 👤 get_user_profile      → 查用户的个人资料和过敏         │ │
│  │ 📈 get_diet_history      → 查用户过去的饮食记录          │ │
│  │ 🏥 get_health_summary    → 获取综合健康报告             │ │
│  │ 🧠 learn_preference      → 学习并记住用户的偏好          │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  📖 技能书(5个技能，按场景激活)：                           │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ 🗣️ conversation       → 始终激活，控制说话风格          │ │
│  │ 🍽️ nutrition-analysis → 用户上传照片或描述食物时激活    │ │

│                                                            │
│  📝 记忆本(5个持久文件 + 3个临时文件)：                       │
```

### 3.5 Agent 之间怎么协作？

> **大白话**：后端有个"调度员"（AgentOrchestrator），它决定什么情况调哪个 Agent。

```
用户拍照上传食物图片时：

  调度员的工作流程：
  ┌────────────────────────────────────────────────────────────┐
  │ 步骤1（同时做）:                                           │
  │   ├── 读用户的记忆文件 → 拿到用户偏好、过敏、疾病         │
  │   └── 直接查数据库 → 拿到用户今天的营养目标和已吃多少     │
  │                                                          │
  │ 步骤2（串行做）:                                           │
  │   └── 调用 nutrition_agent（支持用户记忆） → 拿到食物分析结果     │
  │                                                          │
  │ 步骤3（计算）:                                             │
  │   └── 算一下这餐占今天目标的多少比例                        │
  │                                                          │
  │ 步骤4（合并）:                                             │
  │   └── 把所有结果拼在一起返回给用户                        │
  │                                                          │
  │ 步骤5（后台悄悄做）:                                       │
  │   └── 异步更新用户的记忆文件，让 AI "记住"这次分析         │
  └────────────────────────────────────────────────────────────┘

用户和 Deep Agent 聊天时：

  Deep Agent 自己决定调用哪些工具：
  - 用户问"我今天吃了什么" → 调用 get_diet_history 工具查数据库
  - 用户上传照片 → 调用 analyze_food_image 工具（内部调 nutrition_agent）
  - 用户问"我还能吃什么" → 调用 get_daily_status + query_nutrition_knowledge
  - 用户说"我不喜欢辣的" → 调用 learn_preference 工具记住偏好
```

### 3.6 记忆系统架构

> **大白话**：AI 的记忆就像人写日记——每次对话前先翻翻日记本，写完再记上新的。系统有两套日记本，各有各的写法。

```
┌─────────────── 数据库（真正的账本）─────────────────┐
│ 所有数据都在这里，是"唯一真相源"                       │
│ 用户资料、食物记录、体重、对话……存在 PostgreSQL，图片在 MinIO，缓存在 Redis │
└──────────┬────────────────────────────┬───────────────┘
           │                            │
    ┌──────▼──────┐              ┌──────▼──────┐
    │ 🔁 SyncService │              │ 🧠 Deep Agent │
    │ (定期从DB抄写) │              │ (自己写记忆) │
    └──────┬──────┘              └──────┬──────┘
           │                            │
    ┌──────▼────────────────────────────▼────────────────────┐
    │              📁 每个用户一个文件夹                         │
    │              agents/UserMemory/{用户ID}/                  │
    │                                                          │
    │  📓 日记本一号（SyncService 写的，从数据库同步）：          │
    │  ├── shared/user_memory.md        ← 用户档案和过敏信息    │
    │  ├── goal_tracking/user_goals.md  ← 目标和每日配额        │
    │  ├── nutrition/user_nutrition.md  ← 饮食历史和常吃食物    │
    │  └── chat/user_chat.md            ← 对话偏好和话题        │
    │                                                          │
    │  📓 日记本二号（Deep Agent 自己写的）：                    │
    │  ├── memories/profile.md          ← AI 总结的用户画像      │
    │  ├── memories/goals.md            ← AI 理解的目标         │
    │  ├── memories/nutrition.md        ← AI 观察的饮食趋势      │
    │  ├── memories/preferences.md      ← AI 学到的偏好          │
    │  └── memories/insights.md          ← AI 发现的规律和洞察    │
    │                                                          │
    │  💬 对话记录（Deep Agent 每次对话的检查点）：              │
    │  └── sessions/{线程ID}/checkpoint.md                     │
    └──────────────────────────────────────────────────────────┘
```

**两套记忆会不会冲突？**
会有信息重叠（比如两套都有用户资料和目标），但目前没有自动双向同步。安全网是：
- 日记本一号每天凌晨由 SyncService 从数据库全量重写，保证和数据库一致
- 日记本二号由 AI 自己维护，可能包含 AI 的推理和总结

### 4.2 状态管理架构

> **大白话**：状态管理就是解决"数据变了，界面怎么自动刷新"的问题。Riverpod 是 Flutter 最流行的状态管理方案之一。

```
┌────────────────── Riverpod 工作方式 ───────────────────┐
│                                                       │
│  📦 Provider（数据提供者）                              │
│  ├── 服务型：提供 API 调用服务（如 AuthService）         │
│  ├── 状态型：持有数据并在变化时通知界面（如 authState）   │
│  └── 计算型：从其他数据推导出来（如 isLoggedIn 从authState算出）│
│                                                       │
│  🔄 数据流：                                           │
│  用户操作 → StateNotifier 方法 → ApiService HTTP请求    │
│           → ApiResponse 返回 → 更新 StateNotifier.state│
│           → AsyncValue.when(loading/data/error) → UI 重建│
│                                                       │
│  📊 AsyncValue 的三种状态：                             │
│  ├── loading → 显示加载圈圈                            │
│  ├── data → 显示真实数据                               │
│  └── error → 显示错误信息                              │
└───────────────────────────────────────────────────────┘
```

**为什么用 Riverpod？**
- 类型安全：编译时就能发现数据类型错误
- 可测试：Mock Provider 就能测试，不需要真网络请求
- 依赖注入：服务通过 Provider 创建和注入，不用到处传参数

### 4.3 认证流程

> **大白话**：APP 启动后的认证流程就像进酒店——看看有没有有效房卡，没有就去前台办一张。

```
APP启动
  │
  ▼
初始化 ApiService（全局单例，所有 HTTP 请求用它）
  │
  ▼
初始化路由（watchAuthState 自动监听登录状态变化）
  │
  ▼
检查本地存储的 Token：
  ├── 没有 Token → 跳转到登录页
  ├── 有 Token → 调 GET /auth/verify-token 验证
  │     ├── 有效 → 直接进首页（或引导页，如果没完成引导）
  │     └── 无效 → 清空 Token → 跳转登录页
  │
  ▼
用户登录成功：
  ├── 保存 accessToken + refreshToken 到安全存储
  ├── 更新 authStateProvider → 触发路由重定向 → 进首页
  └── 检查引导状态 → 未完成引导 → 跳转引导页

日常使用中：
  ├── 每次 HTTP 请求自动带 Authorization: Bearer <accessToken>
  ├── 遇到 401 → 自动用 refreshToken 换新 accessToken → 重试原请求
  └── 换token也失败 → 清空 Token → 退回登录页
```

### 4.4 SSE 流式通信架构

> **大白话**：SSE（Server-Sent Events）就像看直播——服务器一边分析，一边告诉你进度，不用等全部分析完才能看到结果。你拍照后，APP 上会显示"正在分析图片..."→"正在提取营养..."→"正在生成建议..."→"完成！"。

```
┌── Flutter APP ──────────────────────────────────────────┐
│                                                          │
│  📸 拍照识别流程：                                       │
│  1. 上传图片到后端（普通 HTTP 请求）                      │
│  2. 创建食物记录（SSE 连接）                             │
│     → 收到 {type: "session"} → 显示"已创建记录"          │
│     → 收到 {type: "status", "正在分析图片"} → 显示进度    │
│     → 收到 {type: "content", step="extract_nutrition"}   │
│        → 显示营养数据                                     │
│     → 收到 {type: "content", step="generate_advice"}     │
│        → 显示建议                                        │
│     → 收到 {type: "complete"} → 显示"分析完成"          │
│                                                          │
│  💬 AI 对话流程：                                         │
│  1. 发送消息（SSE 连接）                                 │
│     → 收到 {type: "content", content="我建议..."}        │
│        → 逐字显示 AI 回答（打字机效果）                  │
│     → 收到 {type: "complete"} → 对话结束                 │
│                                                          │
│  ⚠️ 注意：食物识别用 HttpClient（更精确的行分割），       │
│          AI 对话用 Dio（复用已有的 Auth 拦截器），         │
│          两套实现风格略有不同，但效果一样。                │
└──────────────────────────────────────────────────────────┘
```

### 4.5 缓存架构

> **大白话**：APP 里也有三层缓存，就像你的钱包、抽屉和银行——越近的越快，但容量越小。

```
请求图片数据：
  │
  ├── 第1层：内存缓存（钱包，最快）
  │     最多存 100 条普通数据 + 50 条图片
  │     24小时过期
  │     找到了？直接返回！
  │
  ├── 第2层：SharedPreferences（抽屉，稍慢）
  │     JSON 格式存，24小时过期
  │     找到了？先放进内存缓存，再返回
  │
  ├── 第3层：文件缓存（银行，最慢但最全）
  │     图片 base64 解码后存到临时目录
  │     不过期（依赖系统清理）
  │     找到了？先放进内存缓存，再返回
  │
  └── 都没有？→ 从服务器 API 获取
             → 存入内存缓存 + SharedPreferences + 文件缓存
             → 返回数据
```

## 六、安全架构

### 6.1 认证安全

| 安全措施 | 实现方式 | 说明 |
| --- | --- | --- |
| 密码存储 | bcrypt 12 轮哈希 | 即使数据库泄露，也无法从哈希反推密码 |
| JWT 签名 | HS256 算法 | 防止 Token 被篡改 |
| 双令牌 | access(30min) + refresh(7天) | 短令牌泄露影响有限，长令牌只用来换短令牌 |
| 安全存储 | FlutterSecureStorage（加密） | Token 存在 APP 的安全存储区，不能被其他 APP 读取 |
| 自动刷新 | 401 响应时自动用 refreshToken 换 accessToken | 用户无感知 |
| 认证守卫 | GoRouter redirect + FastAPI 依赖注入 | 前后端双重保护 |

### 6.2 已知安全待改进项

| 项目 | 现状 | 风险等级 | 建议 |
| --- | --- | --- | --- |
| CORS 允许 `*` | 开发环境可接受，生产环境不安全 | 高 | 生产环境限制具体域名 |
| JWT 密钥硬编码 | 默认值在生产环境必须更换 | 高 | 使用强随机密钥，从环境变量读取 |
| API 限流未启用 | RateLimiter 已实现但未使用 | 中 | 在路由中启用限流 |
| 前端 IP 硬编码 | `192.168.1.108` 写在代码里 | 低 | 迁移到环境配置 |
| 无数据库迁移 | 使用 create_all() 无法改表结构 | 中 | 引入 Alembic |

## 八、架构决策与已知限制

### 8.1 关键架构决策

| 决策 | 选择 | 为什么这样选 |
| --- | --- | --- |
| 前后端分离 | Flutter + FastAPI | 前后端独立开发部署，改前端不影响后端 |
| SSE 而非 WebSocket | HTTP SSE | AI 分析是单向推送（服务器→客户端），SSE 更简单、防火墙友好 |
| Markdown 记忆 | 文件而非数据库 | AI 原生理解 Markdown，人类可读可调试，比 JSON 更灵活 |
| 多 Agent 而非单 Agent | 5 个专门 Agent | 职责清晰，各 Agent 独立优化和测试 |
| Deep Agent 用工具直连 DB | 工具内 SessionLocal | 避免多一层 API 调用，性能更好 |
| 双令牌认证 | access + refresh | 安全性比单令牌好，用户体验比每次重新登录好 |
| 不用独立 Redis 缓存 | Redis 承担查询缓存 + 前端本地缓存 | 提升查询性能，支持限流 |
| 不用 MinIO 对象存储 | 图片直存 MinIO | 减少数据库体积，分离静态资源 |
| 三层前端缓存 | 内存 + 本地 + 文件 | 平衡速度和容量，图片缓存命中率最高 |
| 事件驱动更新记忆 | fire_and_forget | 不阻塞用户响应，定时任务做兜底 |
| create_all() 建表 | 启动时自动创建 | 快速原型开发，适合早期项目 |

### 8.2 已知限制与风险

| 限制 | 描述 | 影响 | 建议 |
| --- | --- | --- | --- |
| ⚠️ 无数据库迁移 | 用 `create_all()` 建表，不能改已有表结构 | 加列/改类型需要手动操作数据库 | 引入 Alembic 迁移 |
| ⚠️ 双记忆系统重叠 | MemoryManager 4 工作区 vs Deep Agent 5 记忆文件 | 信息可能不一致 | 统一为一套记忆系统 |
| ⚠️ SSE 实现不统一 | FoodService 用 HttpClient，ChatService 用 Dio | 维护两套代码 | 统一为一套 SSE 实现 |
| ⚠️ 前端服务层重复 | 顶层 services/ 和 feature 内 domain/services/ 功能重叠 | 同功能多份代码 | 合并为统一 Service 层 |
| ⚠️ Deep Agent 直连 DB | 工具内 `SessionLocal()` 无依赖注入 | 连接池可能隐患 | 通过 DI 传入 DB session |
| ⚠️ 无前端缓存失效 | 写操作后不清除相关缓存 | 数据可能短暂不一致 | 实现写后清除策略 |
| ⚠️ IP 硬编码 | `192.168.1.108` 在代码里 | 换开发环境需改代码 | 环境配置化 |
| ⚠️ 无测试套件 | pytest 依赖有了但无测试目录 | 代码质量没有自动验证 | 补充单元/集成测试 |
| ℹ️ 环境变量不一致 | langgraph.json 用 `.env`，后端用 `.env.dev` | 可能导致配置不一致 | 统一环境变量文件 |
